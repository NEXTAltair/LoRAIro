# ADR 0061: 登録パイプライン再設計 — pHash 候補の重複/別版分類

- **関連 Issue**: #630 / #631 / #632 / #633

## Context

現在の画像登録は、pHash が完全一致した時点で「重複確定」とみなし、新規 `images`
レコードを作らず既存 ID を返す (`ImageRepository.add_original_image` /
`ImageDatabaseManager.register_original_image`)。この設計は単純だが、**同一 pHash に
なり得る「別版」を先行画像へ吸収してしまう**。具体的には次の差分が失われる:

- カラー版 / グレースケール版 (見た目の彩度が異なるが pHash は一致しやすい)
- 解像度・サイズ違い
- 透過 (アルファ) 有無の違い

加えて、登録パイプラインには pHash 完全一致前提に依存した構造的課題が残っている:

1. **保存タイミングの責務 (#632)**: `_prepare_image_metadata()` が重複チェックより
   前に `fsm.save_original_image()` を実行する経路がある。重複スキップされる画像でも
   ファイルコピーが残り得る。GUI ワーカーは事前 `detect_duplicate_image()` で軽減して
   いるが、`register_original_image()` を直接呼ぶ経路では保存が先に走る。
2. **副作用の経路間差異 (#633)**: 重複時の関連 `.txt` / `.caption` 取り込み・filename
   alias 登録・統計値 (`registered` / `skipped` / `failed`) の扱いが GUI ワーカー /
   API / direct 呼び出しで揃っていない。

本 ADR は #630 / #631 / #632 / #633 を含む**登録パイプライン再設計全体の設計判断**を
記録する SSoT である。#631 (グレースケール相当判定) はこの再設計の基盤実装にあたる。

## Decision

### 1. pHash 完全一致は「重複確定」ではなく「候補」とする

pHash 完全一致を最終判断にせず、**追加属性比較を経て「重複」か「別版」かを分類する**。
完全に同一と判断できる場合だけ既存画像へ寄せ、属性差が重要なら別版として新規行を作る。

- **ハミング距離による近似重複検知は導入しない**。pHash 完全一致のみを候補抽出の
  起点とする。近似検知はしきい値運用が難しく、別版分類という本 ADR の目的とは別軸。
- **`phash` カラムは unique 化しない**。候補検索キーとして使い、複数行が同一 pHash を
  共有することを許容する。`index` は維持して候補検索を高速化する。

### 2. 分類属性

候補 (同一 pHash) どうしを比較し、重複/別版を判定する属性:

- `width` / `height`
- `has_alpha`
- `is_grayscale_like` (#631 で追加。内容ベースのグレースケール相当判定)

必要に応じて `mode` / `format` / `extension` / `color_space` を補助的に用いる。
`colorfulness_score` は閾値調整・診断用の連続値で、分類の直接条件には用いない。

#### `is_grayscale_like` / `colorfulness_score` の判定方式 (#631)

`mode == "L"` だけでは不十分 (RGB 画像でも全チャンネルがほぼ同じなら実質グレー)。
JPEG 圧縮ノイズで `R == G == B` が崩れるため、完全一致ではなく閾値付きの内容判定にする:

- 画像を RGBA へ変換し、サムネイル化してサンプリングする。完全透過 (不可視) 画素は
  任意の RGB を持ち得るため、彩度算出から除外する (含めると見えない色で誤分類する)。
- 可視画素ごとに `max(|R-G|, |G-B|, |R-B|)` のチャンネル間最大差分を算出する。
- その 99 パーセンタイル (`method="higher"` で実在画素値) を `colorfulness_score` とし、
  閾値以下なら `is_grayscale_like=True` とする。平均ではなく高パーセンタイルを採るのは、
  画像のごく一部にのみ色が乗るケース (ロゴ・透かし・小さなカラー領域) を「カラー」として
  残すため。100 パーセンタイル (= max) は単一外れ値画素に過敏なため避け、99 を採る。
- 縮小は `BILINEAR` (面積保存系) を使い、細い色領域 (罫線・透かし) を取りこぼさない。
  パレットモード (`P` / `1`) は Pillow が縮小時に `BILINEAR` を無視して `NEAREST` 化
  するため、縮小前に RGB へ変換する。透過カラーのにじみは、縮小前の元アルファを最近傍
  参照して可視画素のみ採用するガードで防ぐ。
- 動画像 (アニメ GIF / WebP) は 1 フレーム目のみ評価する。pHash・保存も同フレーム基準で
  整合する。フレーム横断の集約は #631 のスコープ外 (必要なら別版分類 #630 側で扱う)。
- 閾値・パーセンタイル・サンプリングパラメータは `file_system.py` のモジュール定数として
  根拠コメント付きで定義する。

### 3. 保存タイミングは分類後 (#632)

pHash 計算後・保存前に分類できる構造へ見直す。**save は分類結果が「新規」または
「別版」のときだけ実行する**。

- 重複と判定された候補は保存しない (ファイルコピーを残さない)。
- save 済みだが DB 登録しないことが確定した場合の cleanup 規約を定める (保存物を削除し
  孤児ファイルを残さない)。
- GUI / API / direct のいずれの経路でも保存タイミングを揃える。
- 重複時に 512px 生成や関連ファイル処理が必要な場合の順序を明確化する。

### 4. 副作用は分類結果駆動で全経路統一 (#633)

重複時 / 別版時 / 新規時の副作用を一箇所で定義し、GUI / API / direct の登録経路すべてで
同じルールを使う:

- 関連 `.txt` / `.caption` の取り込み先を分類結果に従わせる。
- filename alias 登録の責務と失敗時挙動を統一する。
- 統計値 (`registered` / `skipped` / `failed`) の意味を経路間で揃える。

### 5. スコープ前提

- **1 プロジェクト = 1 DB 前提**。プロジェクト横断での重複/別版判定スコープは対象外。

### 6. backfill 方針 (#631)

`is_grayscale_like` / `colorfulness_score` は NULL 許容で追加し、**既存 DB は遅延
backfill** を既定とする。migration では全件再計算しない (内容サンプリングを伴うため
起動時一括 backfill はコストが高く UX を損なう)。NULL を「未判定」として扱い、再
アノテーションや別版分類など判定が必要になった経路で遅延 backfill する。新規登録は
`FileSystemManager.get_image_info` が値を埋めるため NULL にならない。

## Consequences

- **別版登録で `images` 行が増える**。1 DB = 1 プロジェクト前提のため、この行増加は許容
  範囲。横断スコープを持たないので増加は局所的。
- **`UniqueConstraint(uuid, phash)` は維持できる**。`uuid` は登録ごとにユニークに発番
  されるため、同一 pHash の別版が複数行になっても複合制約には抵触しない。`phash` 単独の
  unique 化はしない (候補検索キーのため)。
- 保存タイミングと副作用が分類結果駆動に統一され、経路依存の不整合 (#632 / #633) が
  解消される。重複スキップ時の孤児ファイルが残らなくなる。
- 分類ロジックが追加されるぶん登録経路の制御フローは複雑化するが、責務が一箇所に集約
  されることで全体の見通しは改善する。

## 実装フェーズ

本 ADR が定める再設計は段階的に実装する。本 ADR 起票時点では基盤 (#631) のみ実装する:

- **基盤 (#631)**: `is_grayscale_like` / `colorfulness_score` カラム追加、`get_image_info`
  での内容判定、`add_original_image` での永続化、Alembic migration。本フェーズ。
- **後続**: 候補分類 (#630) + 保存タイミング見直し (#632) → 副作用統一 (#633)。
